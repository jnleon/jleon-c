Referential integrity ensures that relationships between tables remain consistent. We enforce referential integrity and other rules by applying constraints to our tables.

A constraint is associated with a table and defines properties that the column data must comply with.

Types of Constraints:

  • NOT NULL
  • UNIQUE
  • PRIMARY KEY - allowes FKs to establish a relations, enforces NOT NULL and UNIQUE
  • FOREIGN KEY - enforces valid PK values, limits deletion of PK row if FK row exists
  • CHECK - specifides acceptable values that can entered in a column
  • DEFAULT - provides a default value for the column

When we make multiple INSERT, UPDATE, or DELETE statements and want them to be all or nothing then we would consider it a transaction, or single unit of work. It should be committed on success or rolled back on failure.

The ACID Test determines if a series of actions should be in a transaction

  • Atomicity - all or nothing
  • Consistency - thnigs are in a consistent state at the end
  • Isolation - concurrent execution of a transaction results as if executed serially. (If you all send me money at the same time, my balance should be the total).
  • Durability - once a transaction has been committed, there's no going back even after power loss, crash or error.
